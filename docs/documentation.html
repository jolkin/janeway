Documentation

The user-facing API, the four supported run modes, and the configuration knobs that distinguish them.

Janeway runs as a single Docker container that exposes its full functionality over HTTP. This page documents the run modes most users need (baseline, with visualization, with fault injection, with a ROS 2 bridge) and the complete HTTP API surface.

Prerequisite Janeway is shipped as a Docker image and orchestrated with Docker Compose. If you don't yet have a working clone, follow the installation tutorial first.

Baseline operation ¶

The default invocation runs Janeway as a pure HTTP API. Every endpoint described under API reference is reachable; no browser frontend, fault oracle, or ROS integration is enabled. 

docker compose up

The first invocation builds the image (5–10 minutes); subsequent invocations reuse it. Janeway is ready to accept traffic the moment compose logs: 

Uvicorn running on http://0.0.0.0:8000

Ports exposed

PortService
:8000HTTP API (this is the surface you submit plans to)
:9004Plan visualization (d3-dag of the active plan)

Submit a plan

The three input formats correspond to three endpoints:

# RMPL — Janeway's native format
curl -X POST http://localhost:8000/execute \
     -H "Content-Type: text/plain" \
     --data-binary @my_program.rmpl

# PDDL — converted to a state plan via pddl_to_sp, then planned by Kirk
curl -X POST http://localhost:8000/execute-pddl \
     -F "domain=@my_domain.pddl" \
     -F "problem=@my_problem.pddl" \
     -F "plan_file=@my_pddl_plan.txt"

# Raw state plan — forwarded verbatim to Kirk
curl -X POST http://localhost:8000/execute-state-plan \
     -H "Content-Type: application/json" \
     --data-binary @my_state_plan.json

Each of these returns 202 Accepted with a JSON body of {"status": "dispatched", ...}. Dispatch proceeds asynchronously — to follow what happens next, open a stream against /violations (see below).

Generated artifacts

Every plan run produces a snapshot and a set of service logs under the host-mounted directory generated_plans/:

FileDescription
<timestamp>_<source>.jsonPlan JSON snapshots (RMPL / PDDL / state-plan inputs and Kirk outputs).
janeway.logThe eaas logger plus any Uvicorn output.
kirk-serve.logKirk's Lisp planning server stdout/stderr.
dispatcher.log, monitor.log, local-oracle.log, local-agent.log, telemetry.log, plan-visualization.logPer-service stdout/stderr.

Running with visualizations ¶

Setting ENABLE_VIS=1 additionally starts the Vite-served browser frontend. The plan-visualization graph at :9004 is always on; ENABLE_VIS specifically adds the scenario-aware browser scene (e.g. the drone view). 

ENABLE_VIS=1 docker compose up

Once the container is ready, the browser frontend lives at:

http://localhost:5173

The Vite dev server proxies /ws to Janeway's in-container telemetry server, so the browser reaches both the page and the WebSocket through the same origin. Override VIS_WS_URL only if the visualization is served from a CDN or reverse proxy that can't forward /ws back into the container.

Drone scene — visualization as execution oracle

The drone scene is special-cased: the in-browser visualization also acts as the dispatch oracle. Each time the drone visually completes an action, the frontend POSTs the matching *_end event to the dispatcher and the action's effect to the causal-link monitor. This lets you run the drone example with ENABLE_ORACLE=0 and still get a fully advancing mission.

VerbMonitor state update
flyDRONE1.DRONE-AT = <"to" arg>
scoopDRONE1.HAS-WATER = true, DRONE1.TANK-EMPTY = false
deliverDRONE1.HAS-WATER = false, DRONE1.TANK-EMPTY = true, HOUSEN.FIRE = false, HOUSEN.EXTINGUISHED = true

Two manual fault buttons in the visualization remain available and POST directly to the monitor.

Running with fault injection ¶

Two modes are available, and they compose.

Blanket negation

Set SIMULATE_FAULTS=1. After posting the correct state update for any causal link, the oracle immediately posts a conflicting update for the same variables. Useful as a smoke test for the violation pipeline. 

SIMULATE_FAULTS=1 docker compose up

Spec-driven faults

Mount a YAML file describing which actions should fail, how many times, and what state assignment to inject: 

# faults.yaml
faults:
  - action: drive          # matches event names like DRIVE_END_8, drive_3_end
    times: 2               # fire the fault twice, then stop
    assignment:
      rover1.location: "lost"
  - action: science
    times: 1
    assignment:
      rover1.has_sample: false

The spec file lives on the host. Mount it in through a docker-compose.override.yml (auto-loaded by compose; ignored from version control): 

# docker-compose.override.yml
services:
  eaas:
    volumes:
      - ./faults.yaml:/app/faults.yaml:ro
    environment:
      FAULT_SPEC_FILE: /app/faults.yaml

Then start the container as normal:

docker compose up
Modes compose If both SIMULATE_FAULTS=1 and FAULT_SPEC_FILE are set, an action listed in the spec uses the spec's assignment; every other action falls back to blanket negation.

Running with a ROS 2 bridge ¶

To drive a real robot — or any external system — set ENABLE_ORACLE=0. The local oracle is disabled and the PyKirk dispatcher binds to 0.0.0.0:9000 so execution reports can come in from outside the container. The bundled ros_bridge package is the supported integration; any HTTP client posting to /handle_execution will also work. 

ENABLE_ORACLE=0 docker compose up

What the bridge does

  • Outbound (container → ROS): connects to the in-container telemetry WebSocket and publishes every dispatch event on /eaas/events as a std_msgs/String containing JSON.
  • Inbound (ROS → container): subscribes to /eaas/execution_reports and forwards each message as an HTTP POST to the dispatcher's /handle_execution endpoint.

Install the bridge in your ROS 2 workspace

Requires a sourced ROS 2 workspace (Humble or later) with rclpy and std_msgs.

# From your ROS 2 workspace src/ directory
ln -s /path/to/janeway/ros_bridge .
cd ..
pip install websockets aiohttp
colcon build --packages-select ros_bridge
source install/setup.bash

Run the bridge

# Default — connects to localhost:8002 (telemetry) and localhost:9000 (dispatcher)
ros2 run ros_bridge bridge_node

# Custom URLs
ros2 run ros_bridge bridge_node --ros-args \
  -p telemetry_ws_url:=ws://192.168.1.10:8002/ws \
  -p dispatcher_url:=http://192.168.1.10:9000

# Or via launch file
ros2 launch ros_bridge bridge.launch.py \
  telemetry_ws_url:=ws://192.168.1.10:8002/ws

Bridge parameters

ParameterDefaultDescription
telemetry_ws_urlws://localhost:8002/wsTelemetry WebSocket URL (inside the Janeway container).
dispatcher_urlhttp://localhost:9000Dispatcher HTTP URL (inside the Janeway container).
event_topic/eaas/eventsROS topic for outbound dispatch events.
report_topic/eaas/execution_reportsROS topic for inbound execution reports.
reconnect_delay3.0Seconds to wait before reconnecting after a WS drop.

Sending an execution report from ROS

Publish a std_msgs/String to /eaas/execution_reports with a JSON body:

{
  "event": "drive_1_end",
  "execution_time": 12.5,
  "is_controllable": true
}

Listening for dispatch events

Subscribe to /eaas/events. Each message is a JSON string with the telemetry event format:

{
  "type": "execution",
  "timestamp": "2026-03-23T14:30:00.000",
  "source": "agent",
  "data": {
    "agent_id": "agent_0",
    "event": "drive_1_start",
    "verb": "drive",
    "time": 4.0,
    "controllable": true
  }
}

Configuration reference ¶

All scenario behavior is controlled by environment variables, set either inline before docker compose up or in a .env file next to docker-compose.yml.

VariableDefaultWhen to set it
ENABLE_VIS01 to start the Vite visualization frontend at :5173.
ENABLE_ORACLE10 to disable the local oracle and expose the dispatcher at :9000 for external execution reports.
SIMULATE_FAULTS01 for blanket-negation fault injection from the oracle.
FAULT_SPEC_FILEunsetPath inside the container to a YAML fault spec.
MISSION_STATUS_CALLBACK_URLunsetURL to POST {"status": "completed"|"fail"} when a mission terminates.
VIS_WS_URLderivedOverride only when the browser can't reach the telemetry WebSocket through the Vite proxy.

API reference ¶

POST /execute

Submit an RMPL program for planning and execution.

  • Request body — either raw RMPL text with Content-Type: text/plain, or JSON with Content-Type: application/json and an "rmpl" key.
  • Optional header — X-Package-Name (RMPL package to plan; default main).
  • Response — 202 Accepted with {"status": "dispatched", ...}.
curl -X POST http://localhost:8000/execute \
     -H "Content-Type: text/plain" \
     -H "X-Package-Name: my-package-name" \
     --data-binary @my_program.rmpl

POST /execute-pddl

Submit a PDDL problem for planning and execution. Multipart form data with these fields:

FieldTypeDescription
domainfile uploadPDDL domain file (durative actions).
problemfile uploadPDDL problem file (objects, init, goal).
plantext fieldTemporal plan — one START: action(args) [DUR] line per action. Either plan or plan_file is required.
plan_filefile uploadSame temporal plan, uploaded as a file. Convenient for clients that pipe a planner's output to a file.

The temporal plan format follows the standard PDDL temporal planner output:

0.0: load(package1, truck1, city1) [1.0]
1.0: drive(truck1, city1, city2) [3.0]
4.0: unload(package1, truck1, city2) [1.0]

POST /execute-state-plan

Submit a state plan JSON directly. Skips pddl_to_sp and any RMPL-to-plan translation; the body is forwarded verbatim to Kirk's POST /plan-from-state-plan. 

curl -X POST http://localhost:8000/execute-state-plan \
     -H "Content-Type: application/json" \
     --data-binary @my_state_plan.json

POST /resume

Continue an in-progress mission with an updated plan. Unlike /execute*, which always starts a new mission and resets the causal-link monitor to a blank world state, /resume preserves the monitor's observed current_state across re-initialization. Typical use is after a fault halt: read the live world state from GET /state, hand it to your planner, then POST the resulting state plan back here. 

# Typical resume flow after a fault halt
curl -X GET  http://localhost:8000/state > current_state.json
# ... planner generates a new state plan from current_state.json ...
curl -X POST http://localhost:8000/resume \
     -H "Content-Type: application/json" \
     --data-binary @new_state_plan.json

GET /state

Return the current state of the world as tracked by the causal-link monitor.

curl http://localhost:8000/state
# {"assignments": {"rover1.location": {"variable": "rover1.location", "value": "science1"},
#                  "rover1.has_sample": {"variable": "rover1.has_sample", "value": true}}}

GET /health

Returns the liveness of the server and all internal services.

curl http://localhost:8000/health
# {"status": "ok", "services": {"kirk": "ok", "dispatcher": "ok", "agent": "ok", "oracle": "ok"}}

GET /violations

Server-Sent Events (SSE) stream of plan-execution outcomes — both causal-link violations and terminal mission-status notifications. The connection stays open and delivers events in real time. 

curl -N http://localhost:8000/violations

Violation event (when the monitor detects a causal-link conflict):

{
  "timestamp": "2026-03-26T14:30:00.000000+00:00",
  "source": "state-update",
  "violations": ["True violates active (Q=False): Start->Action2)"]
}

The source field indicates how the violation was detected: "state-update" (oracle state post), "event-observation" (action start/end event), or "event:<timing>:<name>" (telemetry).

Mission-status event (when the dispatcher finishes a plan):

{
  "timestamp": "2026-05-26T20:43:36.300000+00:00",
  "source": "dispatcher:agent_0",
  "status": "completed"
}

status is "completed" when every event executed and "fail" when the dispatcher halted (e.g. due to a violation). Status events don't trigger a halt — they're informational.
