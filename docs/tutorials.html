Tutorials

Install Janeway, run the bundled drone-firefighting PDDL example, and learn how to extend the same recipe to your own plans.

This page walks you through a full first run from a clean machine: clone the repository, build the container, and submit the example PDDL problem under examples/pddl/. It should take about 10–15 minutes the first time, almost all of which is the image build.

1. Install prerequisites ¶

You need:

  • Git (with submodule support — any recent version).
  • Docker with the docker compose plugin (Docker Desktop on macOS / Windows already ships with both; on Linux install Docker Engine + the Compose plugin).
  • cURL — for the example commands below. Any HTTP client works.

Confirm both are reachable from your shell:

docker --version
docker compose version
git --version

2. Clone the repository ¶

Janeway uses git submodules to vendor the Kirk planner, PyKirk dispatcher, PDDL converter, and causal-link monitor. Clone with --recurse-submodules to fetch them all in one shot: 

git clone --recurse-submodules https://github.com/jolkin/janeway.git
cd janeway

Already cloned without submodules? You can hydrate them after the fact: 

git submodule update --init --recursive

3. Build and start the container ¶

From the repository root:

docker compose up

The first invocation builds the image, which takes 5–10 minutes — it compiles the Kirk Lisp planner from source and installs the Python dispatch layer. Subsequent invocations reuse the image and start in seconds.

Janeway is ready to accept traffic when you see this line in the compose logs:

Uvicorn running on http://0.0.0.0:8000

Sanity-check the server in a second terminal:

curl http://localhost:8000/health
# {"status": "ok", "services": {"kirk": "ok", "dispatcher": "ok", "agent": "ok", "oracle": "ok"}}
Tip Leave the docker compose up terminal open so you can watch the server, oracle, and dispatcher logs scroll by as the next step runs.

4. Run the drone PDDL example ¶

The repository ships a small drone-firefighting domain: a drone repeatedly flies between a lake and houses on fire, scooping water and delivering it until the fires are out. The three input files live under examples/pddl/:

FileRole
drone_domain.pddlPDDL domain — types, predicates, and the three durative actions (fly, scoop, deliver).
drone_problem.pddlPDDL problem — drone, lake, and houses; goal is for every house to be extinguished.
drone_plan.txtA pre-generated temporal plan that solves the problem.

The bundled run.sh wraps the right cURL invocation:

cd examples/pddl
./run.sh

# Equivalent to:
# curl -X POST http://localhost:8000/execute-pddl \
#      -F "domain=@drone_domain.pddl" \
#      -F "problem=@drone_problem.pddl" \
#      -F "plan_file=@drone_plan.txt"

If everything is wired up correctly, you'll see a response like:

{"status": "dispatched", "mission_id": "..."}

The compose terminal will start emitting dispatch events as the oracle drives the plan forward — fly_1_start, fly_1_end, scoop_2_start, and so on.

5. Watch the execution ¶

Live event stream

In a third terminal, tail the violation/mission-status SSE stream:

curl -N http://localhost:8000/violations

When the dispatcher finishes, you'll see a terminal status event. For the unmodified drone plan it should be: 

{"timestamp": "...", "source": "dispatcher:agent_0", "status": "completed"}

Plan visualization

Point your browser at:

http://localhost:9004

You'll see a d3-dag rendering of the plan with events, temporal constraints, episode edges, and causal links laid out left-to-right. The view updates as new plans are submitted.

Inspect the current world state

curl http://localhost:8000/state

Browse generated artifacts

Every plan run produces a snapshot under the host-mounted directory generated_plans/ next to docker-compose.yml, alongside the per-service logs (janeway.log, kirk-serve.log, dispatcher.log, etc.).

6. Where to go from here ¶

  • Try the same example with the drone visualization scene by adding ENABLE_VIS=1: 
    ENABLE_VIS=1 docker compose up
    Then open http://localhost:5173.
  • Add a fault to the run — see Running with fault injection for the YAML spec format and the SIMULATE_FAULTS blanket mode.
  • Drive the example from a ROS 2 system instead of the in-container oracle — see Running with a ROS 2 bridge.
  • Swap in your own PDDL domain — supply your three files to the /execute-pddl endpoint exactly the way run.sh does. The API reference documents the multipart form fields.
